# Database Schema Pattern

DB スキーマ設計のパターンと意図。

## テーブル分解パターン

**プロパティは個別テーブルに分割する。**

Nullable なカラムを追加するのではなく、別テーブルに分ける。

## 設計の意図

UPDATE を最小化し、システムの複雑性を削減する。
後で更新したくなるケースは、実は別テーブルが必要なシグナル。

1. **データの投入タイミングで分離**
   - 同時に記録するものだけ同じテーブルに
   - 投入タイミングが異なるものは別テーブルに

2. **条件分岐や NULL をテーブル構造で置き換える**
   - フラグや NULL による分岐は意味が曖昧になりやすい
   - テーブルの存在/非存在で状態を表現すれば意味が一意になる
